# Search & Ranking Systems - Curriculum

How to design search, autocomplete, and ranking systems — common in interviews and every product.

---

## Module 1: Search System Architecture
- [ ] **High-level architecture**: query → query parser → index lookup → ranking → results
- [ ] **Indexing pipeline**: source data → extract → transform → tokenize → build index → serve
- [ ] **Online vs offline**: indexing is offline (batch/stream), serving is online (low latency)
- [ ] **Components**:
  - [ ] **Crawler/Ingester**: fetch or receive source data
  - [ ] **Indexer**: build inverted index from documents
  - [ ] **Query service**: parse query, search index, rank results, return
  - [ ] **Index store**: distributed inverted index (sharded by document or term)
- [ ] **Sharding strategies**: shard by document ID (each shard has all terms for its docs) vs shard by term (each shard has all docs for its terms)
- [ ] **Replication**: replicate each shard for availability and read throughput

## Module 2: Inverted Index Deep Dive
- [ ] **Forward index**: document → list of terms (how documents are stored)
- [ ] **Inverted index**: term → list of documents (how search engines query)
- [ ] **Posting list**: for each term, a sorted list of (doc_id, frequency, positions)
- [ ] **Index construction pipeline**:
  - [ ] Tokenization: split text into tokens ("New York City" → ["new", "york", "city"])
  - [ ] Normalization: lowercase, remove accents, Unicode normalization
  - [ ] Stemming: "running" → "run", "better" → "good" (language-dependent)
  - [ ] Stop words: remove "the", "is", "at" (or keep for phrase queries)
  - [ ] Synonym expansion: "NYC" → "new york city"
- [ ] **Index updates**:
  - [ ] Immutable segments (like LSM SSTables), merged in background
  - [ ] Near real-time search: new documents searchable within seconds (Elasticsearch refresh)
- [ ] **Compression**: posting lists compressed with variable-byte encoding, delta encoding

## Module 3: Ranking & Relevance
- [ ] **The ranking problem**: 10,000 documents match query — which 10 to show first?
- [ ] **Text relevance signals**:
  - [ ] **TF-IDF**: term frequency (how often in THIS doc) × inverse document frequency (how rare across ALL docs)
  - [ ] **BM25**: improved TF-IDF with document length normalization (Elasticsearch default)
  - [ ] Field boosting: title matches worth more than body matches
  - [ ] Phrase matching: "machine learning" ranked higher than "machine" + "learning" apart
- [ ] **Non-text signals** (features):
  - [ ] Freshness: newer content ranked higher (news, social media)
  - [ ] Popularity: click-through rate, view count, engagement
  - [ ] Personalization: user's past behavior, preferences, location
  - [ ] Authority: PageRank-style, domain authority, verified accounts
- [ ] **Learning to Rank (LTR)**: use ML model to combine signals into a relevance score
  - [ ] Pointwise: predict relevance score per document
  - [ ] Pairwise: predict which of two documents is more relevant
  - [ ] Listwise: optimize the entire ranking list
- [ ] **Two-stage ranking**: fast retrieval (inverted index) → expensive re-ranking (ML model) on top-K
- [ ] Design decision: start with BM25, add signals incrementally, add ML ranking when you have click data

## Module 4: Autocomplete & Typeahead
- [ ] **Autocomplete**: suggest completions as user types ("sys" → "system design", "system design interview")
- [ ] **Data source**: past queries, popular searches, trending topics
- [ ] **Trie (prefix tree)**: data structure optimized for prefix lookups
  - [ ] Each node = character, paths from root = prefixes
  - [ ] Store: frequency/score at leaf or internal nodes
  - [ ] Lookup: traverse trie to prefix → return top-K children by score
- [ ] **Optimization**:
  - [ ] Pre-compute top-K at each node (avoid traversing all children at query time)
  - [ ] Trie in memory: fast, but size-limited
  - [ ] Trie sharded by prefix ranges: "a-m" on shard 1, "n-z" on shard 2
- [ ] **Alternative: prefix query on inverted index** (simpler, Elasticsearch prefix/completion suggester)
- [ ] **Ranking autocomplete**: query frequency, recency, personalization, trending boost
- [ ] **Serving**: p99 < 50ms requirement — must be extremely fast
  - [ ] Cache aggressively: most prefixes repeat ("how to" cached for everyone)
  - [ ] Client-side debouncing: don't query on every keystroke (wait 100-200ms)
- [ ] Design decision: trie for dedicated autocomplete service, Elasticsearch prefix for simpler needs

## Module 5: Vector Search & Semantic Retrieval
- [ ] **Keyword search limitation**: "car" doesn't match "automobile" — no semantic understanding
- [ ] **Embeddings**: represent text/images as dense vectors (768-1536 dimensions)
  - [ ] Generated by ML models (BERT, OpenAI, Sentence Transformers)
  - [ ] Semantically similar items have similar vectors (cosine similarity)
- [ ] **Vector search**: find nearest neighbors in embedding space
  - [ ] **Exact (brute force)**: compute distance to every vector — O(n), doesn't scale
  - [ ] **Approximate Nearest Neighbor (ANN)**: trade accuracy for speed
    - [ ] HNSW (Hierarchical Navigable Small World): graph-based, most popular
    - [ ] IVF (Inverted File Index): cluster vectors, search nearest clusters
    - [ ] Product quantization: compress vectors for memory efficiency
- [ ] **Vector databases**: Pinecone, Weaviate, Milvus, Qdrant, pgvector (PostgreSQL)
- [ ] **Hybrid retrieval**: combine keyword search (BM25) + vector search → fuse results
  - [ ] Reciprocal Rank Fusion (RRF): merge two ranked lists
- [ ] Use cases: semantic search, recommendation, RAG (Retrieval Augmented Generation), image search
- [ ] Design decision: keyword search for exact matching, vector search for semantic, hybrid for best results

## Module 6: Search Infrastructure at Scale
- [ ] **Elasticsearch architecture**: cluster → nodes → indexes → shards → segments
  - [ ] Primary shards + replica shards for HA and read scaling
  - [ ] Shard sizing: 10-50GB per shard as guideline
- [ ] **Index lifecycle**: hot (active writes) → warm (less frequent reads) → cold (archival) → delete
- [ ] **Re-indexing**: change mapping/analyzer → must rebuild index (zero-downtime with alias swap)
- [ ] **Search relevance tuning**: A/B test ranking changes, measure click-through rate
- [ ] **Cache layers**: query result cache, filter cache, field data cache
- [ ] **Consistency trade-off**: indexing is near-real-time (~1 second refresh), not instant
  - [ ] "Write then immediately search" may not find the new document
- [ ] **Multi-tenancy in search**: index per tenant vs shared index with tenant filter
- [ ] Monitoring: search latency (p50, p99), indexing throughput, query rate, cache hit ratio
