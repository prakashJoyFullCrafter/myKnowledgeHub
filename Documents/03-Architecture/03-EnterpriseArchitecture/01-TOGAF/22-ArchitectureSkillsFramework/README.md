# Architecture Skills Framework (TOGAF Series Guide) - Curriculum

> **TOGAF Architecture Skills Framework** — defines competencies, role categories, and skill levels for architects. The "what should an architect know, and at what depth, for which role?" answer that promotes architecture to a recognised profession with a career path.

## Module 1: Why a Skills Framework
- [ ] Architect roles are inconsistently defined across orgs: same title, vastly different expectations
- [ ] Symptoms of missing skills framework: architects with title but no practice, hire mistakes, no career progression, "architect = senior developer who attended a course"
- [ ] **TOGAF's purpose**: provide an objective vocabulary for what architects do and at what level — usable for recruiting, performance review, growth plans, certification
- [ ] **Maps to**:
  - [ ] Job descriptions and recruiting (what to look for)
  - [ ] Career frameworks (paths from architect → senior architect → enterprise architect)
  - [ ] Training and certification plans (where to invest learning hours)
  - [ ] Team composition (what mix of skills covers a programme)
- [ ] **Foundational principle**: architecture is a **profession**, not a senior-developer rank — has its own skill domain distinct from coding

## Module 2: Architect Role Categories
- [ ] TOGAF defines (and most orgs adapt) roughly **eight architect role categories**:
  - [ ] **Architecture Board Member** — governance role; reviews and approves architectures
  - [ ] **Architecture Sponsor** — executive funding/political backing
  - [ ] **Architecture Manager** — runs the EA practice
  - [ ] **Enterprise Architect** — owns enterprise-wide architecture vision and roadmap
  - [ ] **Business Architect** — business capabilities, value streams, operating model
  - [ ] **Data Architect** — data structure, governance, master data
  - [ ] **Application Architect** — application portfolio, integration, application landscape
  - [ ] **Technology Architect** — infrastructure, platform, technology standards
- [ ] **Solution Architect** (often added to TOGAF's list) — project/programme-level; bridges enterprise architecture to delivery
- [ ] **Domain Architect** (modern addition) — owns architecture within a bounded business or technical domain (e.g. payments architect, security architect)
- [ ] **Modern variants**:
  - [ ] **Cloud Architect** — overlaps Technology Architect with cloud-platform specialism
  - [ ] **Platform Architect** — owns the internal developer platform (often co-owned with Platform Engineering)
  - [ ] **Integration Architect** — focused on inter-system contracts, ESB/iPaaS, API strategy
- [ ] **Career path** typically: Developer → Tech Lead → Solution/Domain Architect → Enterprise Architect / Chief Architect → CTO/CIO

## Module 3: Skill Categories
- [ ] TOGAF organises skills into seven main categories:
  - [ ] **Generic Skills** — leadership, teamwork, communication, negotiation, customer interaction
  - [ ] **Business Skills & Methods** — business case development, financial management, organisational change, business process consulting
  - [ ] **Enterprise Architecture Skills** — modelling, frameworks (TOGAF/Zachman/etc.), capability building blocks, design principles
  - [ ] **Programme/Project Management Skills** — managing business change, project mgmt, governance
  - [ ] **IT General Knowledge** — software engineering, system development methods, IT industry trends
  - [ ] **Technical IT Skills** — software, application, data, infrastructure, networks (per domain)
  - [ ] **Legal & Regulatory** — contract law, fraud, compliance, data protection
- [ ] **Per category, each skill is rated for proficiency** (see Module 4) for each role
- [ ] **Pattern**: junior architects need depth in Technical IT + EA Skills; senior architects need breadth + heavy Generic + Business + Legal skills
- [ ] The skills framework is comprehensive — most orgs **subset and tailor** it to their context

## Module 4: Proficiency Levels
- [ ] TOGAF defines 4 proficiency levels per skill:
  - [ ] **Level 1 — Background**: aware of the topic; can follow when others discuss it
  - [ ] **Level 2 — Awareness**: understands key concepts; can ask intelligent questions
  - [ ] **Level 3 — Detailed**: works with the skill regularly; competent practitioner
  - [ ] **Level 4 — Expert**: expert; can lead, teach, set strategy in this area
- [ ] **Role-skill matrix**: for each role × each skill, expected level is specified
  - [ ] Example: Enterprise Architect × Modelling → Level 4 (Expert)
  - [ ] Example: Enterprise Architect × Coding → Level 2 (Awareness)
  - [ ] Example: Solution Architect × Coding → Level 3 (Detailed) or even Level 4
- [ ] **Use cases**:
  - [ ] **Hiring**: evaluate candidate against expected levels for role
  - [ ] **Performance review**: identify level gaps with current incumbent
  - [ ] **Personal development**: choose what to invest in learning to progress to next role
  - [ ] **Team gap analysis**: across team, where are we missing Expert-level coverage?

## Module 5: Skills Inventory & Career Pathways
- [ ] **Skills inventory practice**:
  - [ ] Annual self-assessment + manager review against role's expected skills matrix
  - [ ] Identify development priorities (1-2 skills per cycle, not all at once)
  - [ ] Link to learning budget, conference attendance, internal projects
- [ ] **Career pathway design**:
  - [ ] **Vertical**: depth in one domain (Data Architect → Senior Data Architect → Chief Data Officer)
  - [ ] **Horizontal**: breadth across domains (Application → Enterprise Architect → Chief Architect)
  - [ ] **Hybrid**: T-shaped — deep in one, broad in adjacent
- [ ] **Promotion criteria**: not just years served, but evidence of skill-level achievement (Level 3 → Level 4 transitions)
- [ ] **Cross-domain rotation** (powerful but underused): Solution Architect spends 6 months as Business Architect → builds business linkage that pure tech path misses
- [ ] **Mentorship pairing**: Level 4 in skill X mentors Level 2-3 colleagues — accelerates learning, scales expert knowledge

## Module 6: Modern Adaptations & Anti-Patterns
- [ ] **Modern competencies the original TOGAF Skills Framework under-emphasises**:
  - [ ] **Cloud-native architecture** (AWS/Azure/GCP, serverless, containers)
  - [ ] **DevOps & SRE** integration (architecture meets operations)
  - [ ] **Data engineering & ML/AI** architecture (modern data stack)
  - [ ] **Platform engineering & internal developer platforms**
  - [ ] **Sustainability/green architecture**
  - [ ] **Distributed systems theory** (deeper than the original framework's coverage)
  - [ ] **Modern security** (zero-trust, supply chain, DevSecOps)
- [ ] **Pair with**:
  - [ ] **Skills frameworks alternatives**: SFIA (Skills Framework for the Information Age) — UK-origin, widely adopted, more granular than TOGAF's
  - [ ] **Stack Overflow / GitHub competency models**
  - [ ] **Vendor-specific** certifications (AWS Solutions Architect, GCP Professional Cloud Architect, CKA)
- [ ] **Anti-patterns**:
  - [ ] **Skills framework as bureaucracy**: filling cells in a spreadsheet annually with no behaviour change
  - [ ] **Promotion based purely on tenure** ignoring skill demonstration
  - [ ] **One-size-fits-all expectations**: same skill matrix for all "architects" regardless of role
  - [ ] **Title inflation**: "Architect" for senior dev to keep them happy, with no actual change in expectations
  - [ ] **No exit ramp from architect track**: forcing people back to delivery / IC tracks if architecture isn't right for them
  - [ ] **Ignoring soft skills** (Generic Skills category): architects fail far more often on communication and influence than on technical depth
  - [ ] **Overlooking Business + Legal categories**: most senior architects under-invest here, capping career at Solution Architect

## Recommended Practice
| Module | Practice |
|--------|----------|
| Modules 1-2 | Define your org's architect role categories; pick one role and write expected outcomes (not titles) |
| Module 3 | For your current role, score yourself across the 7 TOGAF skill categories — what's the shape? |
| Module 4 | Build a role-skill matrix for one role; map expected levels per skill |
| Module 5 | Identify the 2 skill-level gaps that block your promotion to next role; build a 12-month learning plan |
| Module 6 | Audit your org's architect promotions over last 3 years: were they based on demonstrated skill levels or tenure? |
| Module 6 | Compare TOGAF Skills Framework vs SFIA for one role — which fits your org better? |
