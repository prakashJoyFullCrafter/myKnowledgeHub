# TOGAF 10 Certification Preparation

Complete preparation for TOGAF Foundation (Level 1) and TOGAF Practitioner (Level 2) exams.

---

## Exam Overview

### TOGAF Foundation — Level 1
| Aspect | Detail |
|---|---|
| Questions | 40 multiple-choice |
| Duration | 60 minutes |
| Pass mark | 55% (22/40 correct) |
| Book status | CLOSED book |
| Delivery | Pearson VUE (online or test center) |
| Cost | ~$320 USD |
| Validates | Knowledge and understanding of TOGAF 10 terminology, structure, and core concepts |

### TOGAF Practitioner — Level 2
| Aspect | Detail |
|---|---|
| Questions | 8 complex scenario-based questions (gradient scoring) |
| Duration | 90 minutes |
| Pass mark | 60% |
| Book status | OPEN book (TOGAF Standard only — no notes) |
| Delivery | Pearson VUE |
| Validates | Ability to APPLY TOGAF to real-world scenarios |
| Scoring | Gradient — each answer option has different mark value; partial credit possible |

### Combined Path
- Must pass Level 1 before Level 2 (can take back-to-back same day)
- Credential: **TOGAF® Practitioner** — Credly digital badge (shareable on LinkedIn)
- TOGAF 9.x certifications remain valid; TOGAF 10 is additive

## Practitioner Exam: Scenario Types

The 8 scenario questions test application of TOGAF to realistic situations:

1. **ADM Phase Identification** — "Which phase does this activity belong to?"
2. **Stakeholder Management** — "Who should be involved in this decision and why?"
3. **Governance Decisions** — "What action should the Architecture Board take?"
4. **Architecture Content** — "Which artifact type is most appropriate here?"
5. **Building Blocks** — "Is this an ABB or SBB? Justify."
6. **Enterprise Continuum** — "Where does this reference model sit on the Continuum?"
7. **Change Management** — "How should this change request be handled?"
8. **Repository Management** — "Where in the Architecture Repository does this belong?"

**Open-book strategy:** Know the page numbers for key content. Common lookups:
- ADM phase descriptions
- Six classes of Architecture Repository
- Compliance levels (6)
- ACMM levels
- Enterprise Continuum diagram

## Key Terms Glossary — 40 Must-Know Definitions

| Term | Definition |
|---|---|
| ADM | Architecture Development Method — the step-by-step process for developing and managing enterprise architectures |
| Architecture | A formal description of a system at component level, guiding its implementation |
| Architecture Building Block (ABB) | Defines WHAT is needed — a capability or function, technology-agnostic |
| Architecture Contract | Joint agreement between development partners and sponsors on deliverables, quality, and fitness-for-purpose |
| Architecture Domain | One of BDAT: Business, Data, Application, Technology |
| Architecture Governance | The practice by which enterprise architectures are managed and controlled |
| Architecture Landscape | Architectural assets in use or planned: Strategic, Segment, Capability levels |
| Architecture Principles | Enduring rules that inform how an organization fulfills its mission |
| Architecture Repository | Central holding area for all architecture-related work products |
| Architecture Roadmap | List of work packages to move from Baseline to Target Architecture |
| Architecture Vision | Concise description of the Target Architecture addressing key stakeholder concerns (Phase A output) |
| Baseline Architecture | Current state (as-is) of the enterprise |
| Building Block | A package of functionality defined to meet a business need |
| Business Architecture | Representation of strategy, governance, organization, and key business processes |
| Capability Architecture | Detailed architecture for a specific capability (most granular Architecture Landscape level) |
| Capability Increment | A discrete step in the progression of a capability toward its target state |
| Compliance Level | One of: Irrelevant, Consistent, Compliant, Conformant, Fully Conformant, Non-Conformant |
| Concerns | Stakeholder interests that determine the acceptability of the system |
| Dispensation | Formal exception to an architecture standard, granted by the Architecture Board |
| Enterprise Continuum | Classification mechanism: Foundation → Common Systems → Industry → Organization-Specific |
| Foundation Architecture | Most generic, most reusable level of the Enterprise Continuum (e.g., TRM) |
| Gap Analysis | Technique for identifying differences between Baseline and Target Architecture |
| Governance Log | Record of all governance activity: decisions, compliance assessments, dispensations |
| III-RM | Integrated Information Infrastructure Reference Model — Common Systems Architecture for information sharing |
| Principles Catalog | Catalog of architecture principles applicable to the organization |
| Reference Library | Repository section containing guidelines, templates, patterns, and viewpoints |
| Requirements Management | Central ADM process managing requirements throughout all phases |
| Segment Architecture | Architecture for a specific business segment (medium Architecture Landscape level) |
| Solution Building Block (SBB) | Defines HOW a capability will be delivered — specific product or implementation |
| Standards Information Base (SIB) | Database of approved technology standards: mandatory, recommended, emerging |
| Statement of Architecture Work | Formal document defining scope and approach for an architecture engagement |
| Stakeholder | Individual or organization with interests in the architecture outcome |
| Strategic Architecture | Long-horizon enterprise direction (highest Architecture Landscape level) |
| Target Architecture | Desired future state of the enterprise |
| TOGAF Fundamental Content | Stable core of TOGAF 10: ADM, Content Framework, Enterprise Continuum, Repository, Governance, Capability |
| TOGAF Series Guide | Topic-specific extension to TOGAF 10 (Security, Business Architecture, Digital, Agile, etc.) |
| TRM | Technical Reference Model — Foundation Architecture taxonomy of generic platform services |
| Transition Architecture | Intermediate stable state between Baseline and Target |
| View | Architecture representation from a specific stakeholder perspective |
| Viewpoint | Template/specification for constructing a view |
| Work Package | Set of actions to achieve one or more architecture objectives |

## 60 Practice Questions

### Section A — Fundamentals (Q1–15)

**Q1.** What is the primary purpose of the TOGAF ADM?
- a) Provide a vendor-neutral technology reference model
- b) Provide a tested, repeatable process for developing enterprise architectures
- c) Define the organizational structure of the architecture function
- d) Classify architecture assets in the Enterprise Continuum

**Answer: B** — ADM is the core process engine of TOGAF.

---

**Q2.** Which best describes the Enterprise Continuum?
- a) A sequential process for architecture development
- b) A repository for storing architecture work products
- c) A classification mechanism for architecture assets from generic to organization-specific
- d) A governance framework for architecture compliance

**Answer: C** — Enterprise Continuum classifies assets; the Repository stores them.

## Common Exam Traps

| Trap | The Correct TOGAF Answer |
|---|---|
| "Architecture" = drawings | Architecture = formal description of BDAT structure (4 domains) |
| "Compliance" = binary pass/fail | 6 compliance levels; Non-Conformant AND Irrelevant are both valid outcomes |
| ADM phases are always sequential | ADM explicitly supports iteration within and across phases |
| Enterprise Continuum stores artifacts | Continuum = CLASSIFIES; Repository = STORES |
| Artifact and Deliverable are interchangeable | Deliverables are formally signed off; artifacts are components within them |
| TRM is a product catalog | TRM is a taxonomy/service classification — no vendor or product names |
| ABB and SBB are interchangeable | ABB = what (tech-agnostic capability); SBB = how (specific product) |
| Architecture Board approves IT budgets | Board approves architecture decisions, standards, dispensations — not budgets |
| Requirements Management is Phase A | It is a CENTRAL HUB process connecting ALL phases, not a sequential phase |
| TOGAF 10 invalidates TOGAF 9 certifications | TOGAF 9.x certifications remain valid |
