# Domain-Driven Design (DDD)

Eric Evans' approach to tackling complexity in software by aligning code with the business domain.

## Sections
- **01. Strategic DDD** - Bounded Context, Context Mapping, Ubiquitous Language
- **02. Tactical DDD** - Aggregates, Entities, Value Objects, Domain Events, Repositories
- **03. Integration Patterns** - Shared Kernel, ACL, Customer/Supplier, Conformist, Open Host Service
- **04. Event Storming** - Collaborative modeling workshop technique

## Why DDD
- Complex domain logic is the hardest part of software
- Aligns code with the language the business actually speaks
- Makes implicit knowledge explicit
- Provides a strategic toolkit for breaking up monoliths and microservice boundaries

## Required Reading
- *Domain-Driven Design* - Eric Evans (the Blue Book)
- *Implementing Domain-Driven Design* - Vaughn Vernon (the Red Book)
- *Domain-Driven Design Distilled* - Vaughn Vernon (short intro)
